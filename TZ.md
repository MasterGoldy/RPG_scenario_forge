# Техническое задание на разработку проекта
# RPG Scenario Forge

## 1. Введение

### 1.1. Наименование проекта
**RPG Scenario Forge** — минималистичная система создания и управления сценариями для настольных ролевых игр.

### 1.2. Цель проекта
Разработка базового веб-приложения, позволяющего мастерам RPG создавать, хранить и просматривать игровые сценарии в сжатые сроки (5 дней разработки).

### 1.3. Целевая аудитория
- Мастера настольных ролевых игр (D&D, Pathfinder, Warhammer и др.)
- Новички в гейм-мастеринге, нуждающиеся в простом инструменте

## 2. Требования к функционалу

### 2.1. Основные модули

#### 2.1.1. Модуль пользователей
- Базовая регистрация и авторизация
- Личный кабинет для управления своими сценариями

#### 2.1.2. Модуль сценариев
- Создание/редактирование/удаление собственных сценариев
- Просмотр списка всех сценариев
- Детальный просмотр сценария
- Базовая фильтрация по автору

#### 2.1.3. Модуль базового анализа
- Статичные советы по улучшению сценариев
- Простая статистика по количеству сценариев

## 3. Структура базы данных

### 3.1. Основные сущности

#### 3.1.1. Пользователь (User)
Стандартная модель Django User для хранения учетных данных пользователей.

#### 3.1.2. Сценарий (Scenario)
Основная сущность приложения. Содержит:
- Название сценария
- Описание и полный текст
- Уровень сложности (начальный, средний, продвинутый)
- Оценочное время прохождения
- Рекомендуемое количество игроков
- Ссылку на автора (пользователя)
- Статус (черновик/опубликован/архив)
- Даты создания и обновления

#### 3.1.3. Категории (Category)
Простая система категорий для организации сценариев по типам:
- Подземелья
- Квесты
- Социальные взаимодействия
- Босс-файты

### 3.2. Схема связей
- Один пользователь может создавать много сценариев
- Каждый сценарий имеет только одного автора

## 4. Роли пользователей

### 4.1. Гость (неавторизованный пользователь)
**Разрешения:**
- Просмотр опубликованных сценариев
- Регистрация в системе

**Ограничения:**
- Не может создавать сценарии
- Не может оставлять отзывы

### 4.2. Авторизованный пользователь
**Разрешения:**
- Все разрешения гостя
- Создание/редактирование/удаление собственных сценариев
- Просмотр своих черновиков
- Изменение статуса сценариев (черновик/опубликовано)

**Ограничения:**
- Не может редактировать или удалять чужие сценарии
- Не может управлять чужими профилями
- Нет прав администратора

### 4.3. Администратор (только разработчик)
**Разрешения:**
- Полный доступ к админ-панели Django
- Управление всеми сценариями при необходимости
- Доступ к базе данных

## 5. Логика работы системы

### 5.1. Создание сценария
1. Авторизованный пользователь нажимает "Создать сценарий"
2. Заполняет основные поля: название, описание, полный текст
3. Выбирает параметры: сложность, время прохождения, количество игроков, категорию
4. Сохраняет как черновик или сразу публикует

### 5.2. Просмотр сценариев
- На главной странице отображаются все опубликованные сценарии
- Можно фильтровать по автору через поиск
- При клике на сценарий открывается детальная страница с полным текстом

### 5.3. Базовый анализ
- На странице анализа отображаются статичные советы по улучшению сценариев
- Показывается простая статистика: количество созданных сценариев
- Нет сложной аналитики или динамических рекомендаций

## 6. Технические требования

### 6.1. Фронтенд
- Адаптивный дизайн на основе Bootstrap
- Минимальная стилизация для экономии времени
- Работа в современных браузерах

### 6.2. Бэкенд
- Django в качестве основного фреймворка
- SQLite для разработки (можно заменить на PostgreSQL при деплое)
- Базовая структура проекта с разделением на приложения

### 6.3. Безопасность (базовый уровень)
- Защита от CSRF атак
- Проверка авторизации для редактирования сценариев
- Базовая валидация форм

## 7. Этапы разработки (5 дней)

### День 1: Настройка проекта и базовая структура
- Инициализация Django проекта
- Настройка базы данных (SQLite)
- Создание базовых моделей: User (стандартная), Scenario, Category
- Настройка админ-панели

### День 2: Аутентификация и CRUD для сценариев
- Реализация регистрации и авторизации
- Создание форм для добавления/редактирования сценариев
- Базовые шаблоны для списка и деталей сценариев

### День 3: Шаблоны и интерфейс
- Создание всех HTML шаблонов
- Интеграция Bootstrap для базового дизайна
- Настройка навигации между страницами

### День 4: Функционал анализа и дополнительные страницы
- Создание страницы анализа со статичными советами
- Реализация страницы подтверждения удаления
- Добавление фильтрации по автору
- Настройка прав доступа (только свои сценарии)

### День 5: Тестирование и финальные правки
- Тестирование основных сценариев использования
- Исправление ошибок
- Базовая оптимизация
- Подготовка к деплою

## 8. Ограничения и упрощения

### 8.1. Что НЕ реализовано в MVP:
- Система тегов (только категории)
- Сложная аналитика (только статичные советы)
- Роли модераторов
- Система оценок и отзывов
- Экспорт/импорт данных
- API для интеграций

### 8.2. Упрощения для ускорения разработки:
- Использование стандартной модели User Django
- SQLite вместо PostgreSQL для разработки
- Статичные CSS без препроцессоров
- Минимальная JavaScript функциональность
- Нет кеширования
- Нет очередей задач
